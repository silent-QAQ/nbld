# 技能系统设计文档

本文档独立承载技能系统、技能编辑器、技能编写读取规范、技能引擎、技能行与技能效果标准库设计。玩法主文档只保留系统入口与关联说明。

## 1. 技能系统设计

本项目的技能系统必须服务当前已经确定的玩法框架：

- 无职业
- 主手 + 副手武器体系
- 技能魂石装配
- 天赋树强化技能
- 固定伤害公式结算

因此技能系统不能做成传统职业技能树，而应做成：

```text
武器动作模板 + 技能魂石配置 + 技能运行时逻辑 + 天赋修正 + 属性结算
```

技能系统的核心目标是：

- 让策划可低成本组合新技能
- 让武器、魂石、天枢能稳定联动
- 让程序运行时结构清晰可维护
- 让客户端表现与服务端判定可对齐

---

## 2. 技能系统总结构

建议将技能系统拆成六层：

### 2.1 技能定义层

负责定义技能本身是什么。

包括：

- 技能 ID
- 技能名称
- 技能类型
- 槽位类型
- 武器适配标签
- 资源消耗
- 冷却时间
- 目标选择方式
- 效果列表

### 2.2 技能装配层

负责定义技能如何进入玩家战斗栏。

包括：

- 主手技能槽装配
- 副手技能槽装配
- 防具被动魂石装配
- 预设切换

### 2.3 技能运行时层

负责技能释放过程中的状态推进。

包括：

- 输入检测
- 前摇
- 施法
- 生效帧
- 后摇
- 可取消窗口
- 命中判定
- 冷却进入

### 2.4 效果执行层

负责技能命中后实际产生什么结果。

包括：

- 伤害
- 位移
- 护盾
- 回复
- 控制
- 异常
- 召唤
- 追加伤害
- Buff / Debuff

### 2.5 修正层

负责所有外部系统对技能的修正。

包括：

- 武器模板修正
- 副手联动修正
- 天赋树修正
- 被动魂石修正
- 装备词条修正
- 场景 Buff 修正

### 2.6 表现层

负责客户端技能表现。

包括：

- 动画
- 特效
- 音效
- 镜头震动
- 飘字
- 受击反馈
- UI 冷却显示

## 3. 技能分类设计

为了便于编辑器和运行时处理，技能建议分为以下类型。

## 3.1 按槽位分类

- `主手主动技能`
- `副手主动技能`
- `防具被动技能`

其中：

- 主手主动技能负责主要输出循环
- 副手主动技能负责辅助、防御、位移、反制、联动
- 防具被动技能负责常驻和条件触发

## 3.2 按释放方式分类

- `瞬发`
- `引导`
- `蓄力`
- `持续区域`
- `位移攻击`
- `锁定释放`
- `开关型`

## 3.3 按命中方式分类

- `近战碰撞`
- `扇形范围`
- `直线穿透`
- `圆形区域`
- `投射物`
- `锁定目标`
- `自身光环`

## 3.4 按效果性质分类

- `伤害类`
- `控制类`
- `增益类`
- `减益类`
- `回复类`
- `位移类`
- `召唤类`
- `异常类`
- `护盾类`

## 4. 技能运行时状态机

技能运行时建议统一使用状态机。

标准流程：

```text
待机
 -> 输入校验
 -> 释放前检查
 -> 前摇
 -> 生效
 -> 后摇
 -> 冷却
 -> 待机
```

### 4.1 输入校验

校验内容：

- 是否在技能栏中装配
- 是否满足武器条件
- 是否满足副手条件
- 是否不在公共禁用状态
- 是否不在冷却
- 是否资源足够
- 是否不在被控制状态

### 4.2 释放前检查

释放前额外处理：

- 扣除资源
- 锁定目标
- 记录施法朝向
- 记录当前武器状态
- 读取天赋和被动修正

### 4.3 前摇阶段

职责：

- 播放起手动画
- 处理起手位移
- 进入霸体或减伤状态
- 开启可取消窗口

### 4.4 生效阶段

职责：

- 生成碰撞体
- 发射投射物
- 施加 Buff / Debuff
- 触发伤害或控制
- 调用追加伤害公式

### 4.5 后摇阶段

职责：

- 播放收招动画
- 判断是否允许闪避取消
- 判断是否允许连段转接

### 4.6 冷却阶段

职责：

- 进入冷却计时
- 接收天赋、被动、命中事件对冷却的修正

## 5. 技能数据结构设计

为了方便编辑器和运行时共用，建议把技能拆成以下配置块。

## 5.1 技能主表

每个技能魂石建议包含以下字段：

- `skill_id`
- `skill_name`
- `skill_desc`
- `rarity`
- `slot_type`
- `skill_category`
- `weapon_tags`
- `offhand_tags`
- `unlock_condition`
- `cooldown`
- `mana_cost`
- `stamina_cost`
- `cast_time`
- `recovery_time`
- `can_cancel`
- `cancel_window_start`
- `cancel_window_end`
- `target_type`
- `range`
- `angle`
- `radius`
- `projectile_speed`
- `projectile_lifetime`
- `base_animation`
- `hit_vfx`
- `cast_sfx`
- `effect_group_id`

## 5.2 效果子表

一个技能可以包含多个效果项。

每个效果项建议包含：

- `effect_id`
- `skill_id`
- `effect_type`
- `trigger_timing`
- `target_filter`
- `damage_type`
- `base_ratio`
- `flat_damage`
- `extra_ratio`
- `buff_id`
- `debuff_id`
- `control_type`
- `control_duration`
- `move_distance`
- `shield_ratio`
- `heal_ratio`
- `stack_count`
- `limit_count`
- `condition_group_id`

## 5.3 条件子表

用于描述技能效果触发条件。

建议字段：

- `condition_group_id`
- `condition_type`
- `param_1`
- `param_2`
- `logic_operator`

示例条件：

- 命中数量 >= 3
- 目标生命比例 < 0.35
- 攻击方持有指定 Buff
- 目标存在寒霜层数 >= 4
- 副手类型 = 轻盾

## 5.4 修正子表

用于描述武器、天枢、被动魂石对技能的修正。

建议字段：

- `modifier_id`
- `source_type`
- `source_id`
- `target_skill_id`
- `modify_field`
- `modify_mode`
- `modify_value`
- `apply_condition_group_id`

## 7. 技能效果类型定义

为了防止策划直接写死复杂逻辑，技能效果应标准化为若干效果类型。

第一版推荐内置以下效果类型：

- `DealDamage`
- `DealExtraDamage`
- `ApplyBuff`
- `ApplyDebuff`
- `ApplyControl`
- `ApplyDot`
- `ApplyShield`
- `Heal`
- `MoveCaster`
- `MoveTarget`
- `SpawnProjectile`
- `SpawnArea`
- `SummonUnit`
- `ReduceCooldown`
- `RestoreMana`
- `RestoreStamina`
- `ConsumeBuff`
- `AddStack`
- `DetonateStack`

规则：

- 技能效果必须由标准效果节点组合而成
- 不鼓励第一版大量写单技能专属硬编码逻辑
- 特殊逻辑通过 `条件 + 效果 + 修正` 组合实现

## 8. 技能释放与判定规则

### 8.1 客户端职责

客户端负责：

- 输入响应
- 播放动画与特效
- 本地预览范围
- 本地非权威表现
- 冷却和资源预显示

### 8.2 服务端职责

服务端负责：

- 技能释放合法性校验
- 真实命中判定
- 真实伤害结算
- Buff / Debuff 状态变更
- 控制状态下发
- 冷却与资源权威同步

### 8.3 本地预测建议

第一版建议只做有限预测：

- 本地立即播技能动作
- 本地预显示命中特效
- 真正伤害与控制结果以服务端为准

这样可以降低同步复杂度。

## 9. 技能编辑器设计

技能编辑器的目标不是给程序自己用，而是让策划、美术、技术策划能共同维护技能内容。

编辑器需要解决的问题：

- 技能数量增加后如何低成本新增技能
- 如何避免策划直接改错字段
- 如何让技能效果、条件、标签、表现分离
- 如何快速验证技能可装配范围和运行逻辑

建议将编辑器做成以下模块。

---

## 10. 技能编辑器模块划分

## 10.1 技能列表页

功能：

- 浏览所有技能魂石
- 按主手 / 副手 / 被动分类筛选
- 按武器标签筛选
- 按天枢标签筛选
- 按稀有度筛选
- 按关键字搜索

建议展示字段：

- 技能名称
- 技能 ID
- 技能类型
- 适配武器
- 适配槽位
- 冷却
- 消耗
- 最近修改人
- 最近修改时间

## 10.2 技能基础信息页

用于编辑：

- 名称
- 描述
- 技能图标
- 技能类别
- 武器适配标签
- 副手适配标签
- 释放方式
- 目标类型
- 冷却
- 法力消耗
- 耐力消耗

## 10.3 技能效果编排页

用于配置技能具体效果序列。

推荐采用“时间线 + 效果块”形式。

每个效果块可配置：

- 触发时机
- 效果类型
- 范围形状
- 目标筛选
- 数值参数
- 绑定动画事件

示例：

- `0.10s`：前移 `1.5m`
- `0.25s`：生成扇形碰撞
- `0.25s`：DealDamage `1.2`
- `0.25s`：ApplyDebuff `slow`
- `0.40s`：允许闪避取消

## 10.4 条件逻辑页

用于配置技能中的条件分支。

示例：

- 如果目标生命 < 35%，追加执行 `DealExtraDamage`
- 如果副手 = 轻盾，则附加 `GuardBuff`
- 如果已激活 `破军天枢核心`，则开放穿透

建议使用可视化条件树，而不是让策划手写脚本。

## 10.5 联动修正页

用于展示和编辑：

- 武器模板修正
- 副手联动修正
- 天赋天枢修正
- 被动魂石修正
- 装备词条修正

这页的目标是让策划明确看到：

- 哪些系统正在修改这个技能
- 修改了哪个字段
- 最终运行时会怎么叠加

## 10.6 表现配置页

用于绑定：

- 动画
- 特效
- 音效
- 命中闪光
- 屏幕震动
- UI 图标
- 施法条表现

表现层必须与数值层分离，避免以后调整倍率时误改特效逻辑。

## 10.7 测试预览页

这是技能编辑器里最重要的一页之一。

建议支持：

- 选择主手武器
- 选择副手武器
- 选择技能槽装配
- 选择天赋预设
- 选择目标假人属性
- 实时预览命中、冷却、资源、伤害

如果没有这页，策划会很难独立验证技能结果。

---

## 11. 技能编辑器推荐交互

建议技能编辑器采用以下交互方式：

### 11.1 左侧列表 + 中央配置 + 右侧预览

- 左侧：技能列表
- 中间：基础配置 / 效果时间线 / 条件树
- 右侧：运行结果预览、标签、联动摘要

### 11.2 参数控件标准化

常用字段应使用专用控件：

- 百分比字段用百分比输入框
- 时间字段用秒输入框
- 范围字段用米制输入框
- 标签字段用多选标签面板
- 条件字段用可视化逻辑块

不要让策划直接编辑原始 JSON。

### 11.3 错误提示

编辑器必须内置校验。

例如：

- 没有适配武器标签
- 没有目标类型
- 没有效果块
- 伤害技能没有配置伤害类型
- 命中逻辑与表现事件时间不一致
- 主手技能被错误配置到副手槽位

---

## 12. 技能编辑器数据校验

建议至少做以下自动校验。

## 12.1 静态校验

- 必填字段完整
- 数值范围合法
- 冷却不为负
- 消耗不为负
- 范围参数不为负
- 标签必须存在于标签库
- 引用的 Buff / Debuff / 特效资源必须存在

## 12.2 逻辑校验

- 没有命中行为却配置了伤害
- 追加伤害链超过限制
- 技能标签与武器标签不兼容
- 核心天赋修正的目标技能不存在
- 条件分支永远不可触发

## 12.3 平衡校验

建议增加软性报警：

- 技能倍率过高
- 冷却过短
- 资源消耗过低
- 多个修正叠加后超出建议区间

---

## 13. 技能编辑器与资源管线

技能编辑器不应只管数值，还应纳入资源管线。

建议每个技能关联：

- 图标资源
- 动画资源
- 特效资源
- 音效资源
- 文本描述
- 命中音效
- 预览假人模板

编辑器应支持：

- 一键检查资源缺失
- 一键导出技能配置
- 一键生成预览包

## 14. 技能配置产物建议

建议编辑器最终导出为结构化配置，而不是直接写死进代码。

推荐产物：

- `skills.json`
- `skill_effects.json`
- `skill_conditions.json`
- `skill_modifiers.json`
- `skill_tags.json`

如果项目后期规模扩大，也可以改为：

- 技能主表
- 效果子表
- 条件子表
- 标签表
- 资源映射表

## 15. 技能示例配置思路

以长枪主手技能魂石 `突进穿刺` 为例：

### 15.1 基础定义

- 技能 ID：`skill_spear_dash_001`
- 技能类型：主手主动技能
- 武器标签：`spear`
- 释放方式：位移攻击
- 冷却：`8s`
- 耐力消耗：`15`

### 15.2 效果编排

- `0.00s`：进入前摇
- `0.10s`：前移 `3.5m`
- `0.18s`：生成直线穿刺碰撞
- `0.18s`：DealDamage，物理，倍率 `1.35`
- `0.18s`：ApplyDebuff，短时破甲
- `0.30s`：进入后摇

### 15.3 天赋联动

- 破军核心激活时：命中可穿透
- 流影关键节点激活时：连续命中缩短冷却
- 镇岳路线下：命中减速目标时额外增伤

这就是编辑器中应该能完整可视化看到的内容。

## 16. 第一版技能编辑器落地建议

如果是第一版原型，不要一开始做成过重的全能编辑器。

建议分两阶段：

### 阶段一

- 技能列表页
- 基础信息页
- 效果编排页
- 标签系统
- JSON 导出
- 基础校验

### 阶段二

- 条件树
- 联动修正页
- 预览战斗页
- 平衡报警
- 资源联动检查

这样更符合原型开发节奏。

## 17. 技能系统与编辑器总结

本项目的技能系统最终应固定为：

- 技能来源于 `技能魂石`
- 武器决定动作模板与装配范围
- 副手决定补充能力与联动修正
- 天赋树决定技能强化与机制变化
- 被动魂石决定常驻与条件触发效果

本项目的技能编辑器最终应具备：

- 可视化配置技能基础数据
- 可视化编排技能效果时间线
- 可视化配置条件和联动修正
- 可视化预览技能运行结果
- 可导出结构化配置供客户端和服务端共用

这样才能支撑后续大量新增技能魂石、新武器组合和新天枢内容。

## 18. 技能编写规范

在真正开始做技能编辑器之前，必须先固定“技能如何编写”的统一规范。

如果没有统一规范，后续会出现：

- 同类技能写法不一致
- 字段命名混乱
- 客户端和服务端理解不一致
- 策划难以复用已有技能模板
- 编辑器无法稳定校验

因此第一版必须约束：

- 文件组织规范
- 字段命名规范
- 数值书写规范
- 效果编排规范
- 条件表达规范
- 资源引用规范

---

## 19. 技能配置文件组织规范

建议所有技能配置统一放在独立目录中，例如：

```text
config/
  skills/
    skill_main.json
    skill_effects.json
    skill_conditions.json
    skill_modifiers.json
    skill_tags.json
```

如果项目希望更便于单技能维护，也可以采用“单技能文件 + 汇总导出”模式：

```text
config/
  skills/
    active/
      skill_spear_dash_001.json
      skill_staff_frost_ring_001.json
    passive/
      passive_guard_counter_001.json
```

推荐策略：

- 策划维护时使用“单技能文件”
- 构建导出时汇总成“总表 JSON”

这样兼顾：

- 可读性
- 版本管理
- 运行时加载效率

---

## 20. 技能 ID 规范

每个技能必须使用全局唯一 ID。

推荐格式：

```text
skill_<weapon_or_group>_<theme>_<index>
```

示例：

- `skill_spear_dash_001`
- `skill_staff_frost_ring_001`
- `skill_grimoire_guard_001`
- `skill_offhand_dagger_chase_001`

被动技能建议使用：

```text
passive_<theme>_<index>
```

示例：

- `passive_block_counter_001`
- `passive_burn_boost_001`

规则：

- 全部使用小写字母
- 单词使用下划线分隔
- 不允许中文 ID
- 不允许同一 ID 在不同文件重复出现

---

## 21. 字段命名规范

字段命名统一使用 `snake_case`。

示例：

- `skill_id`
- `mana_cost`
- `cast_time`
- `damage_type`

禁止混用：

- `skillId`
- `SkillID`
- `manaCost`

原因：

- 便于 JSON、Go、Unity、工具链统一映射
- 降低导表和反序列化复杂度

---

## 22. 数值书写规范

为了避免策划与程序对数值理解不一致，技能配置中的数值表达必须固定。

## 22.1 百分比规范

所有百分比一律使用小数表示：

- `5%` 写为 `0.05`
- `12%` 写为 `0.12`
- `150%` 写为 `1.50`

禁止混写：

- `5`
- `5%`
- `0.5%`

## 22.2 时间规范

所有时间字段统一使用秒：

- `0.2`
- `1.5`
- `8.0`

## 22.3 距离规范

所有距离字段统一使用米：

- `1.5`
- `3.0`
- `8.5`

## 22.4 角度规范

所有角度统一使用度数：

- `45`
- `90`
- `180`

## 22.5 层数规范

层数与次数字段统一使用整数：

- `1`
- `3`
- `5`

---

## 23. 技能主表编写规范

每个技能主配置至少必须包含以下字段：

- `skill_id`
- `skill_name`
- `skill_desc`
- `rarity`
- `slot_type`
- `skill_category`
- `weapon_tags`
- `offhand_tags`
- `cooldown`
- `mana_cost`
- `stamina_cost`
- `cast_time`
- `recovery_time`
- `target_type`
- `base_animation`
- `effect_group_id`

其中：

- `weapon_tags` 允许为空数组，但不能缺字段
- `offhand_tags` 允许为空数组，但不能缺字段
- `cooldown`、`mana_cost`、`stamina_cost` 没有就写 `0`
- `skill_desc` 即使暂时为空，也必须保留字段

示例：

```json
{
  "skill_id": "skill_spear_dash_001",
  "skill_name": "突进穿刺",
  "skill_desc": "向前突进并对直线路径上的目标造成物理伤害。",
  "rarity": "rare",
  "slot_type": "mainhand_active",
  "skill_category": "dash_attack",
  "weapon_tags": ["spear"],
  "offhand_tags": [],
  "cooldown": 8.0,
  "mana_cost": 0,
  "stamina_cost": 15,
  "cast_time": 0.10,
  "recovery_time": 0.30,
  "target_type": "forward_line",
  "base_animation": "anim_spear_dash_01",
  "effect_group_id": "eg_skill_spear_dash_001"
}
```

---

## 24. 技能效果编写规范

每个技能至少应包含一个效果项。

一个效果项只做一件事，不要把多个效果硬塞进同一条记录。

正确做法：

- 一条效果做位移
- 一条效果做伤害
- 一条效果做减速

不推荐做法：

- 一条效果里同时写位移、伤害、减速、护盾

这样会让编辑器难做、调试困难、逻辑不透明。

每条效果必须包含：

- `effect_id`
- `skill_id`
- `effect_type`
- `trigger_timing`
- `target_filter`

如果是伤害类效果，还必须包含：

- `damage_type`
- `base_ratio`
- `flat_damage`

如果是控制类效果，还必须包含：

- `control_type`
- `control_duration`

示例：

```json
{
  "effect_id": "ef_skill_spear_dash_001_hit",
  "skill_id": "skill_spear_dash_001",
  "effect_type": "DealDamage",
  "trigger_timing": "0.18",
  "target_filter": "enemy",
  "damage_type": "physical",
  "base_ratio": 1.35,
  "flat_damage": 0,
  "condition_group_id": ""
}
```

---

## 25. 条件编写规范

技能条件必须标准化，不允许策划手写脚本判断。

每个条件组建议表达为：

- 一组条件
- 一组逻辑关系
- 一个最终结果

示例：

```json
{
  "condition_group_id": "cg_low_hp_execute",
  "condition_type": "target_hp_ratio_lt",
  "param_1": 0.35,
  "param_2": 0,
  "logic_operator": "and"
}
```

规范要求：

- 条件 ID 全局唯一
- 条件语义必须清晰
- 条件名优先描述业务含义，而不是实现细节

推荐命名：

- `cg_low_hp_execute`
- `cg_frost_stack_full`
- `cg_offhand_buckler_only`

不推荐命名：

- `condition_01`
- `test_a`
- `new_skill_check`

---

## 26. 修正编写规范

修正是技能系统最容易混乱的部分，必须严格规范。

修正只能修改已有字段，不允许直接在修正表中定义一整个新技能逻辑。

允许的修正类型建议固定为：

- `add`
- `mul`
- `override`
- `append_tag`
- `remove_tag`

示例：

```json
{
  "modifier_id": "mod_pojun_core_spear_dash_pierce",
  "source_type": "talent_core",
  "source_id": "tianqu_pojun_core",
  "target_skill_id": "skill_spear_dash_001",
  "modify_field": "can_pierce",
  "modify_mode": "override",
  "modify_value": true,
  "apply_condition_group_id": ""
}
```

修正规则：

- 武器模板修正优先描述基础动作修正
- 天赋修正优先描述机制与数值修正
- 被动魂石修正优先描述条件触发修正

---

## 27. 标签编写规范

标签必须来自统一标签库，不允许自由手写。

例如：

- 正确：`spear`
- 正确：`buckler`
- 正确：`dash`
- 错误：`spear_like`
- 错误：`fast_dash`
- 错误：`冰冻系`

标签规则：

- 标签名固定
- 标签语义固定
- 新增标签必须先入标签表，再允许被技能引用

这样可以避免：

- 重复标签
- 近义标签混乱
- 编辑器筛选失效

---

## 28. 技能描述文案规范

技能描述不应直接抄配置字段，而应遵循统一文案规则。

建议结构：

- 第一段写核心动作
- 第二段写主要效果
- 第三段写特殊条件或附加说明

示例：

```text
向前突进并穿刺前方敌人，造成物理伤害。
命中护甲目标时附加破甲效果。
激活破军核心后可穿透目标。
```

文案规则：

- 先写玩家感知到的内容
- 后写机制补充
- 避免把内部实现字段直接暴露给玩家

---

## 29. 技能读取规范

“读取”要明确三件事：

- 技能配置如何加载
- 技能配置如何组装成运行时对象
- 客户端和服务端如何读取同一份技能定义

建议统一使用：

```text
静态配置读取 -> 校验 -> 建立索引 -> 生成运行时模板 -> 实例化技能
```

---

## 30. 技能加载流程

建议启动流程如下：

### 30.1 读取原始配置

按顺序读取：

1. `skill_tags.json`
2. `skill_main.json`
3. `skill_effects.json`
4. `skill_conditions.json`
5. `skill_modifiers.json`

先读标签表是为了让后续字段校验有统一参照。

### 30.2 静态校验

静态校验阶段检查：

- JSON 格式是否合法
- ID 是否重复
- 必填字段是否缺失
- 标签是否存在
- 引用的效果组是否存在
- 引用的条件组是否存在
- 修正目标技能是否存在

### 30.3 建立索引

加载后建立以下索引：

- `skill_id -> skill_main`
- `skill_id -> effects[]`
- `condition_group_id -> conditions[]`
- `target_skill_id -> modifiers[]`
- `weapon_tag -> skills[]`
- `slot_type -> skills[]`

### 30.4 生成运行时模板

运行时不直接拿原始 JSON 工作，而应生成不可变模板对象，例如：

- `SkillTemplate`
- `SkillEffectTemplate`
- `SkillConditionTemplate`
- `SkillModifierTemplate`

模板对象职责：

- 供客户端展示
- 供服务端判定
- 供编辑器预览

### 30.5 实例化技能

角色进入战斗时，再根据装配情况实例化技能对象：

- `SkillTemplate` 是静态模板
- `SkillInstance` 是角色当前携带的运行时实例

实例中可保存：

- 当前冷却
- 当前层数
- 当前临时修正
- 当前持有者
- 当前可用状态

---

## 31. 客户端读取规则

客户端读取技能配置的目标是：

- 展示技能信息
- 进行输入绑定
- 播放技能表现
- 做本地有限预测

客户端应读取：

- 技能主表
- 技能效果表中与表现相关的字段
- 标签表
- 资源映射表

客户端不必把所有服务端专用字段全部展开到高频运行逻辑中，但需要保留兼容字段，以便调试和展示。

建议客户端缓存：

- 技能基础展示缓存
- 动画 / 特效资源引用缓存
- 技能模板只读缓存

---

## 32. 服务端读取规则

服务端读取技能配置的目标是：

- 权威校验释放是否合法
- 权威计算命中和伤害
- 权威应用 Buff / Debuff / 控制

服务端必须读取完整技能定义：

- 主表
- 效果表
- 条件表
- 修正表
- 标签表

服务端不依赖客户端下发的技能数值，只依赖：

- 客户端发来的技能 ID
- 施法请求参数
- 角色当前装备状态
- 当前天赋与 Buff 状态

服务端再根据本地模板执行技能。

---

## 33. 编辑器读取规则

编辑器读取技能配置的目标与客户端、服务端不同。

编辑器应支持：

- 读取原始单技能文件
- 读取导出后的总表
- 对比修改前后差异
- 重新生成索引
- 预览合并后运行结果

编辑器需要同时看到两种形态：

- `原始可编辑配置`
- `编译后运行时模板`

这样策划才能明确：

- 自己写了什么
- 程序实际会怎么读

---

## 34. 技能装配读取规则

玩家进入角色或切换预设时，技能装配读取建议按以下顺序执行：

1. 读取主手武器类型
2. 读取副手武器类型
3. 读取主手技能槽装配的魂石 ID
4. 读取副手技能槽装配的魂石 ID
5. 校验魂石是否兼容当前主手 / 副手
6. 读取防具被动魂石
7. 读取当前天赋预设
8. 生成当前战斗技能栏
9. 计算所有技能实例的初始修正结果

若某个技能不兼容当前武器，应：

- 不进入战斗栏
- 标记配置错误
- 在编辑器和开发日志中报错

---

## 35. 技能修正读取顺序

修正顺序必须固定，否则前后端结果会不一致。

建议固定为：

1. 读取技能模板基础值
2. 应用主手武器模板修正
3. 应用副手联动修正
4. 应用天赋节点修正
5. 应用被动魂石修正
6. 应用装备词条修正
7. 应用临时 Buff / Debuff 修正

这样能保证：

- 武器是底盘
- 天赋是成长主修正
- 被动魂石与装备做外围补强
- 临时状态最后生效

---

## 36. 技能缓存规范

为了避免每次释放技能都重复查表，建议建立两层缓存。

### 36.1 静态模板缓存

启动后加载一次：

- 技能模板
- 条件模板
- 修正模板
- 标签模板

只读，不在战斗中修改。

### 36.2 角色运行时缓存

角色进入世界后生成：

- 当前技能栏
- 技能实例状态
- 当前修正结果缓存
- 冷却状态
- 临时层数状态

当以下情况发生时重建或局部刷新：

- 更换主手
- 更换副手
- 更换技能魂石
- 更换天赋预设
- 上下线

---

## 37. 技能热更新与版本规范

如果项目后续支持热更新，技能配置必须带版本号。

建议字段：

- `config_version`
- `skill_schema_version`

作用：

- 识别配置是否与当前客户端兼容
- 识别编辑器导出格式是否过期
- 支持线上灰度检查

技能读取时若发现版本不兼容，应：

- 拒绝加载
- 输出明确错误日志
- 在编辑器中提示需要重新导出

---

## 38. 技能读取错误处理规范

技能读取不能静默失败，必须明确报错。

建议错误级别：

### 38.1 致命错误

直接中止加载：

- 技能主表 ID 重复
- 技能引用不存在的效果组
- 标签表缺失
- JSON 结构损坏

### 38.2 阻断错误

单技能不可用，但系统可继续运行：

- 技能缺少关键字段
- 技能装配标签非法
- 条件组不存在
- 修正目标不存在

### 38.3 警告错误

允许加载，但输出警告：

- 描述为空
- 表现资源未绑定
- 冷却或倍率超出建议范围

---

## 39. 第一版推荐技能编写与读取方案

如果现在要先做第一版，推荐固定为：

- 策划维护单技能 JSON
- 构建时导出为总表 JSON
- 客户端、服务端、编辑器共用同一套导出格式
- 启动时统一静态校验并建立索引
- 战斗时基于模板生成技能实例
- 修正顺序固定为：武器 -> 副手 -> 天赋 -> 被动魂石 -> 装备词条 -> 临时 Buff

这套方案的优点是：

- 开发成本低
- 结构清晰
- 易于做编辑器
- 易于前后端统一

---

## 40. 技能编写与读取总结

技能系统要稳定，先决条件不是编辑器炫不炫，而是规范先定死。

本项目建议先固定：

- 技能 ID、字段命名、百分比、时间、距离的统一写法
- 技能主表、效果表、条件表、修正表的职责边界
- 客户端、服务端、编辑器共享同一份技能配置产物
- 统一的加载、校验、索引、模板化、实例化流程

这样后续不管你做：

- 技能编辑器
- 服务端战斗判定
- 客户端技能栏
- 技能魂石内容扩展

都不会反复返工。

---

## 41. 技能引擎设计

技能系统描述的是“技能有什么内容”，技能引擎解决的是“这些内容如何真正跑起来”。

对于当前项目，技能引擎必须满足以下前提：

- 支持主手 + 副手武器体系
- 支持技能魂石驱动技能内容
- 支持天赋、被动魂石、装备词条对技能的修正
- 支持客户端表现与服务端权威判定分离
- 支持后续大规模新增技能，而不是每个技能都手写逻辑

因此，技能引擎不应写成“每个技能一个脚本”的模式，而应设计成：

```text
配置驱动 + 模板实例化 + 状态机推进 + 效果执行器 + 修正链 + 事件总线
```

---

## 42. 技能引擎职责边界

技能引擎只负责“技能运行”本身，不负责所有战斗系统。

技能引擎负责：

- 接收技能释放请求
- 校验技能是否可释放
- 生成技能运行实例
- 推进技能状态机
- 执行技能效果
- 处理技能命中事件
- 应用技能修正链
- 通知其他系统技能事件

技能引擎不直接负责：

- 角色移动总逻辑
- AI 决策
- 背包与掉落
- UI 逻辑
- 地图同步

这些系统只通过接口与技能引擎交互。

---

## 43. 技能引擎核心模块

建议技能引擎拆成以下模块。

## 43.1 SkillEngine

技能引擎总入口。

职责：

- 管理技能系统生命周期
- 驱动每帧更新
- 协调各子模块
- 维护角色技能实例

## 43.2 SkillTemplateRegistry

技能模板注册表。

职责：

- 保存静态技能模板
- 提供按 `skill_id` 查询
- 提供按标签和槽位筛选

## 43.3 SkillAssembler

技能装配器。

职责：

- 根据主手、副手、魂石、天赋预设生成角色技能栏
- 生成运行时 `SkillInstance`
- 计算初始修正结果

## 43.4 SkillCastValidator

释放校验器。

职责：

- 校验冷却
- 校验资源
- 校验武器兼容性
- 校验控制状态
- 校验距离、目标、朝向

## 43.5 SkillRuntimeController

运行时控制器。

职责：

- 推进技能状态机
- 驱动前摇、生效、后摇
- 处理可取消窗口
- 处理连段转接

## 43.6 SkillEffectExecutor

效果执行器。

职责：

- 执行伤害、控制、位移、Buff、异常等标准效果
- 调用伤害公式
- 调用 Buff / Debuff 系统
- 调用投射物和区域生成系统

## 43.7 SkillModifierResolver

修正解析器。

职责：

- 按固定顺序应用武器、副手、天赋、被动魂石等修正
- 生成技能最终运行参数
- 支持静态修正和动态修正

## 43.8 SkillEventBus

技能事件总线。

职责：

- 广播技能开始、命中、结束、暴击、格挡、追加伤害等事件
- 给天赋、被动、任务、成就等系统订阅

## 43.9 SkillDebugService

技能调试服务。

职责：

- 输出技能运行日志
- 输出命中明细
- 输出修正前后字段变化
- 支持战斗回放排查

---

## 44. 技能引擎运行时对象模型

建议固定三类核心对象。

## 44.1 SkillTemplate

静态模板对象。

特点：

- 来源于技能配置
- 不在战斗中修改
- 客户端和服务端都可持有

包含：

- 基础配置
- 效果配置
- 条件配置
- 静态标签

## 44.2 SkillInstance

角色当前携带的运行时技能实例。

特点：

- 绑定具体角色
- 保存冷却和临时状态
- 可根据当前修正重新生成运行参数

包含：

- 持有者 ID
- 模板引用
- 当前冷却
- 当前可用状态
- 当前层数
- 当前动态修正缓存

## 44.3 SkillExecutionContext

单次施法上下文。

特点：

- 每次释放技能时临时创建
- 释放结束后销毁

包含：

- 施法者
- 当前目标
- 当前朝向
- 当前主手
- 当前副手
- 当前技能实例
- 释放时快照属性
- 修正结果快照
- 事件追踪 ID

这三层对象的职责必须分清：

- `SkillTemplate` 是“技能定义”
- `SkillInstance` 是“角色当前带了什么技能”
- `SkillExecutionContext` 是“这一次技能释放发生了什么”

---

## 45. 技能引擎执行流水线

技能引擎的执行流程建议固定如下：

```text
输入请求
 -> 释放校验
 -> 构建执行上下文
 -> 计算修正结果
 -> 进入状态机
 -> 按时间线执行效果
 -> 触发命中与事件
 -> 进入冷却
 -> 释放结束
```

展开后可分为 10 步：

1. 玩家或 AI 发出技能释放请求
2. `SkillCastValidator` 校验请求是否合法
3. 从 `SkillInstance` 生成 `SkillExecutionContext`
4. `SkillModifierResolver` 计算本次释放的最终参数
5. `SkillRuntimeController` 将技能置入前摇状态
6. 到达生效时机后调用 `SkillEffectExecutor`
7. 命中系统返回命中结果
8. 伤害系统和 Buff 系统处理结果
9. `SkillEventBus` 广播技能事件
10. `SkillInstance` 进入冷却并回到待机状态

## 46. 技能状态机固定设计

技能引擎内部建议固定以下状态：

- `Idle`
- `PreCast`
- `Casting`
- `Active`
- `Recovery`
- `Cooldown`
- `Interrupted`
- `Finished`

### 46.1 Idle

待命状态。

### 46.2 PreCast

释放前摇状态。

处理：

- 起手动画
- 起手位移
- 起手霸体

### 46.3 Casting

施法进行状态。

处理：

- 引导计时
- 蓄力累积
- 输入锁定

### 46.4 Active

技能效果生效状态。

处理：

- 生成碰撞
- 投射物发射
- 区域创建
- 伤害与控制触发

### 46.5 Recovery

后摇状态。

处理：

- 收招动画
- 可取消判断
- 连段转接判断

### 46.6 Cooldown

冷却状态。

### 46.7 Interrupted

被打断状态。

需要定义：

- 是否返还部分资源
- 是否进入完整冷却
- 是否保留蓄力结果

### 46.8 Finished

一次完整释放结束状态。

## 48. 技能效果执行机制

为了避免技能逻辑失控，效果执行器应只识别标准效果类型。

执行机制建议如下：

### 48.1 EffectFactory

按 `effect_type` 生成效果执行单元。

### 48.2 EffectRunner

执行单个效果单元。

### 48.3 EffectBatch

同一时点的多个效果组成一个批次。

例如：

- `0.18s` 同时执行位移结束、伤害、减速

### 48.4 EffectResult

每次效果执行后返回标准结果：

- 是否成功
- 命中数量
- 伤害值
- 是否暴击
- 是否触发异常
- 是否触发额外效果

这样可以让事件系统和日志系统直接消费结果。

- `QueryTargetsByShape`
- `CheckLineHit`
- `SpawnProjectile`
- `SpawnAreaEffect`
- `ApplyDamage`
- `ApplyBuff`
- `ApplyControl`

## 50. 技能修正链设计

修正链是这套玩法的核心差异来源，必须在引擎层固定。

建议分为两类：

## 50.1 静态修正

在角色装配完成后即可计算：

- 主手武器模板修正
- 副手模板修正
- 已激活天赋修正
- 被动魂石常驻修正

这些修正适合缓存。

## 50.2 动态修正

在每次施法时根据上下文计算：

- 目标血量条件

- 当前是否闪避成功后
- 当前是否格挡成功后

## 55. 技能引擎性能原则

技能引擎一定要考虑规模问题，尤其是 MMO 场景。

建议原则：

- 技能模板只加载一次
- 运行时对象池化
- 命中查询走统一接口

- 事件对象尽量复用
- 日志默认分级输出

第一版不需要过早极致优化，但必须避免明显的结构性浪费。

技能引擎不是“技能表读取器”，而是战斗技能在运行时的真实执行内核。

- 状态机控制器
- 效果执行器
- 修正解析器

这样后续再做：

- 技能编辑器

## 58. 技能行系统设计

在当前项目中，除了技能主表、效果表、条件表这类结构化配置外，还可以定义一层更贴近策划编写习惯的 `技能行`。

技能行的定位是：

- 作为单条技能逻辑的最小表达单元
- 作为技能编辑器内部可视化编排的基础语义层
- 作为配置导出前的中间描述格式

可以把它理解为：

> 一条技能行，描述一次“在什么情况下，对什么目标，触发什么效果”。

---

## 59. 技能行包含的核心要素

你目前提出的技能行可携带以下要素，这个方向是正确的。

第一版建议正式固定为：

- `技能名`（必要）
- `目标选择器`
- `触发器`
- `条件`
- `血量要求`
- `触发几率`
- `技能效果`

其中只有：

- `技能名`
- `技能效果`

是核心必备要素。

其余要素可以按需要出现。

---

## 60. 技能行书写规则

技能行采用单行描述，规则固定如下：

- 各要素之间使用 `1 个空格` 分隔
- 各要素之间没有前后逻辑优先级关系
- 解析时通过“要素类型”识别，不通过位置识别
- 同一类要素在一条技能行中通常只允许出现一次

这意味着：

- 技能行是“无序要素组合”
- 不是固定位置参数

例如：

```text
突进穿刺 目标=前方直线 条件=命中 触发几率=1.0 效果=造成物理伤害1.35
```

和：

```text
突进穿刺 效果=造成物理伤害1.35 条件=命中 目标=前方直线 触发几率=1.0
```

在语义上应视为等价。

---

## 61. 技能行语义规则

虽然技能行内各要素没有书写先后关系，但在语义上仍然有职责划分。

建议固定解释方式：

- `技能名`：这条技能行归属于哪个技能
- `目标选择器`：这条技能行的效果作用于谁
- `触发器`：这条技能行何时被触发
- `条件`：这条技能行额外需要满足什么判断
- `血量要求`：对施法者或目标的生命条件限制
- `触发几率`：满足条件后是否按概率生效
- `技能效果`：最终执行什么内容

因此，一条技能行的底层逻辑可以理解为：

```text
当 触发器 发生时，
若 条件 / 血量要求 满足，
则按 触发几率 判定，
然后对 目标选择器 选中的目标执行 技能效果。
```

---

## 62. 技能名规则

`技能名` 是技能行中的必要要素。

职责：

- 作为技能行所属技能的逻辑归类名
- 供策划阅读和检索
- 映射到真正的 `skill_id`

规范：

- 配置层可使用中文展示名
- 导出层必须映射到唯一 `skill_id`
- 同一技能的多条技能行应共享同一个 `skill_id`

示例：

- `突进穿刺`
- `霜环`
- `格挡反震`

---

## 63. 目标选择器设计

`目标选择器` 用于描述技能效果的目标范围。

这是技能行中的重要可选要素。

第一版建议支持以下目标选择器类型：

- `自身`
- `当前目标`
- `前方单体`
- `前方直线`
- `前方扇形`
- `圆形范围`
- `周围敌人`
- `周围友方`
- `命中目标`
- `命中目标周围`

推荐书写方式：

```text
目标=自身
目标=前方直线
目标=命中目标周围
```

规则：

- 若省略目标选择器，则默认取技能主配置中的默认目标类型
- 若技能行显式声明目标选择器，则覆盖默认目标类型

---

## 64. 触发器设计

`触发器` 用于描述这条技能行在什么时机被执行。

第一版建议支持以下标准触发器：

- `释放时`
- `前摇开始`
- `前摇结束`
- `命中时`
- `暴击时`
- `击杀时`
- `闪避后`
- `格挡后`
- `受击时`
- `持续期间每秒`
- `效果结束时`

推荐书写方式：

```text
触发=命中时
触发=暴击时
触发=格挡后
```

### 64.1 技能配置内触发器限制

你提出的这条规则需要固定下来：

> 若处在技能配置内则无法添加触发器

建议正式解释为：

- 技能主配置中已经定义了技能的基础执行时间线
- 处于该基础时间线内的技能行，不允许再额外声明 `触发器`
- 因为其触发时机已经由技能时间线决定

换句话说：

- `技能主动作线中的技能行`：不写 `触发器`
- `被动触发线 / 条件追加线 / 天赋修正线中的技能行`：可以写 `触发器`

这样可以避免：

- 时间线触发和逻辑触发重复定义
- 一个技能行出现双重触发来源

---

## 65. 条件设计

`条件` 用于描述技能行生效前需要额外满足的限制。

第一版建议支持：

- `命中目标`
- `目标带有指定异常`
- `目标处于减速状态`
- `目标生命低于阈值`
- `施法者持有指定 Buff`
- `副手类型符合要求`
- `已激活指定天枢`
- `连击数达到要求`

推荐书写方式：

```text
条件=目标减速
条件=目标灼烧
条件=副手为轻盾
条件=已激活破军核心
```

规则：

- 一条技能行允许 0 到多个条件
- 多个条件之间默认使用 `且`
- 若需要 `或` 关系，应由编辑器内部拆成条件组

---

## 66. 血量要求设计

你单独提出 `血量要求`，这是合理的，因为它在 RPG 技能设计里出现非常频繁，可以视作一类高频条件。

建议把血量要求从普通条件中拆成独立要素。

第一版建议支持：

- `目标血量低于`
- `目标血量高于`
- `自身血量低于`
- `自身血量高于`

推荐书写方式：

```text
血量=目标<0.35
血量=自身<0.50
血量=目标>0.70
```

解释规则：

- `0.35` 表示 `35%`
- 统一使用小数
- 默认比较对象是生命比例，而不是绝对生命值

---

## 67. 触发几率设计

`触发几率` 用于描述技能行在满足其他条件后，是否按概率执行。

推荐书写方式：

```text
几率=1.0
几率=0.25
几率=0.60
```

规则：

- 默认值为 `1.0`
- 若省略，则视为必定触发
- 必须使用 `0.0 ~ 1.0` 的小数表示

示例：

- `几率=1.0` -> 100% 触发
- `几率=0.25` -> 25% 触发

---

## 68. 技能效果设计

`技能效果` 是技能行中的必要要素。

技能效果决定这条技能行最终做什么。

第一版建议支持以下效果表达：

- `造成物理伤害`
- `造成法术伤害`
- `造成追加伤害`
- `附加减速`
- `附加冻结`
- `附加灼烧`
- `回复法力`
- `回复耐力`
- `获得护盾`
- `位移`
- `缩减冷却`
- `叠加层数`
- `引爆层数`

推荐书写方式：

```text
效果=造成物理伤害1.35
效果=附加减速0.20@2.0
效果=获得护盾0.06@4.0
效果=回复法力0.03
```

建议约定：

- `@` 后面表示持续时间
- 伤害后的小数表示倍率
- 护盾、回复等比例默认按最大值比例或配置约定值解释

---

## 69. 技能行标准格式建议

虽然各要素之间没有前后关系，但为了提高可读性，建议编辑器显示和策划书写时优先使用以下推荐顺序：

```text
技能名 目标选择器 触发器 条件 血量要求 触发几率 技能效果
```

这只是推荐显示顺序，不是解析顺序。

示例：

```text
破军追击 目标=命中目标 触发=命中时 血量=目标<0.35 几率=1.0 效果=造成追加伤害0.8
```

---

## 70. 技能行示例

### 70.1 普通主动技能行

```text
突进穿刺 目标=前方直线 效果=造成物理伤害1.35
```

解释：

- 技能名：突进穿刺
- 目标：前方直线
- 无显式触发器，说明这是技能主动作线中的基础效果
- 最终效果为一次物理伤害

### 70.2 低血追击技能行

```text
破军追击 目标=命中目标 触发=命中时 血量=目标<0.35 几率=1.0 效果=造成追加伤害0.8
```

解释：

- 命中后触发
- 仅对低血量目标生效
- 必定触发
- 执行一次追加伤害

### 70.3 控制联动技能行

```text
镇岳压制 目标=命中目标 条件=目标减速 几率=1.0 效果=伤害加成0.08
```

解释：

- 对已减速目标生效
- 提供额外压制收益

### 70.4 冰霜冻结技能行

```text
玄霜凝滞 目标=命中目标 条件=寒霜层数>=4 几率=1.0 效果=附加冻结@1.5
```

解释：

- 目标寒霜层数达标才生效
- 触发一次冻结控制

## 71. 技能行解析规则

技能行解析时建议按“要素类型匹配”而不是“位置匹配”。

解析步骤：

1. 先识别第一个未带前缀的字符串为 `技能名`
2. 再识别带有 `目标=` 的字段
3. 再识别带有 `触发=` 的字段
4. 再识别带有 `条件=` 的字段
5. 再识别带有 `血量=` 的字段
6. 再识别带有 `几率=` 的字段
7. 再识别带有 `效果=` 的字段
8. 将结果组装成 `SkillLineNode`

建议解析结果结构：

- `skill_name`
- `target_selector`
- `trigger`
- `conditions[]`
- `hp_requirement`
- `proc_chance`
- `effect`

---

## 72. 技能行与技能配置的关系

技能行不是要替代完整技能配置，而是作为更高层、更易写的表达层。

建议关系如下：

- `技能主表`：定义技能基础属性
- `技能行`：定义技能逻辑语义
- `效果表/条件表/修正表`：定义程序最终执行结构

也就是说：

```text
技能行 -> 编译/解析 -> 效果表 + 条件表 + 修正表
```

这样技能行可以成为：

- 策划编写入口
- 编辑器可视化语义层
- 导出工具的中间格式

---

## 73. 技能行编辑器限制建议

为了避免技能行失控，编辑器应增加以下限制：

### 73.1 必填限制

- 必须有 `技能名`
- 必须有 `技能效果`

### 73.2 唯一限制

- `目标选择器` 每行最多一个
- `触发器` 每行最多一个
- `血量要求` 每行最多一个
- `触发几率` 每行最多一个

### 73.3 上下文限制

- 若该技能行属于技能基础时间线，则禁止填写 `触发器`
- 若该技能行属于被动触发逻辑，则允许填写 `触发器`

### 73.4 语义限制

- 纯伤害效果不能没有目标
- 概率必须在 `0.0 ~ 1.0` 范围内
- 血量要求只能比较生命比例
- 技能效果必须来自标准效果库

---

## 74. 第一版推荐技能行方案

如果第一版要快速落地，建议直接固定为：

- 技能行支持 7 类要素
- 技能名与技能效果必填
- 要素之间用单空格隔开
- 解析按类型，不按顺序
- 技能基础时间线中的技能行不允许写触发器
- 技能行最终编译为结构化技能效果配置

这套方案兼顾了：

- 策划书写效率
- 程序解析稳定性
- 编辑器实现难度

## 75. 技能行系统总结

技能行系统本质上是在技能配置和技能编辑器之间增加一层“语义描述层”。

它的价值在于：

- 技能表达更接近策划思维
- 要素清晰，便于拆解
- 可解析为统一的运行时结构
- 适合后续做可视化技能编排工具

如果后续继续细化，下一步最值得补的是：

- `技能效果标准语法`
- `技能行编译到效果表的转换规则`
- `技能行编辑器界面结构`

---

## 76. 技能行编译规则

技能行本身不是运行时直接执行的数据结构，它必须先被编译成程序可读取的结构化配置。

本项目中，技能行编译的目标固定为三类产物：

- `效果表`
- `条件表`
- `修正表`

因此，技能行编译的本质是：

```text
技能行语义
 -> 解析要素
 -> 识别上下文
 -> 生成标准效果节点
 -> 生成条件组
 -> 必要时生成修正项
```

---

## 77. 技能行编译输入与输出

### 77.1 编译输入

一条技能行在编译前至少包含：

- `skill_id`
- `skill_line_id`
- `line_context`
- `line_text`

其中：

- `skill_id`：该技能行归属的技能
- `skill_line_id`：该技能行自己的唯一 ID
- `line_context`：该技能行所在上下文
- `line_text`：策划编写的技能行文本

### 77.2 编译输出

一条技能行编译后可能生成：

- `0 到多条效果表记录`
- `0 到多条条件表记录`
- `0 到多条修正表记录`

注意：

- 不是每条技能行都会生成修正表
- 主动作线中的基础效果通常生成效果表
- 条件触发线通常生成条件表 + 效果表
- 天赋/被动改写线通常生成修正表

---

## 78. 技能行上下文分类

技能行在编译前必须先判断自己属于哪种上下文，因为这决定它应该编译到哪类表。

第一版建议固定三种上下文：

### 78.1 主动作线

表示：

- 这条技能行属于技能基础时间线的一部分
- 它是技能本体效果

编译特点：

- 不允许显式写 `触发器`
- 主要生成 `效果表`
- 条件存在时生成 `条件表 + 效果表`

### 78.2 条件触发线

表示：

- 这条技能行属于技能本体之外的追加逻辑
- 需要在某个事件发生时才触发

编译特点：

- 可以写 `触发器`
- 通常生成 `条件表 + 效果表`

### 78.3 修正规则线

表示：

- 这条技能行不是直接执行技能效果
- 而是用于改写某个已有技能效果

编译特点：

- 主要生成 `修正表`
- 条件存在时生成 `条件表 + 修正表`

---

## 79. 要素到数据表的映射规则

技能行中的每个要素，不是平均分配到三个表，而是按职责映射。

固定映射如下：

### 79.1 技能名

用途：

- 映射到 `skill_id`
- 不直接进入效果表、条件表、修正表字段

### 79.2 目标选择器

默认去向：

- `效果表.target_filter`
- 或 `效果表.target_selector`

若当前技能行属于修正规则线，则：

- 目标选择器通常不进入修正表
- 除非它本身是被修改字段

### 79.3 触发器

默认去向：

- `效果表.trigger_timing`

若当前技能行处于主动作线，则：

- 触发时机由技能主时间线决定
- 不从技能行读取 `触发器`

若当前技能行处于修正规则线，则：

- `触发器` 通常转为 `修正表.apply_condition_group_id` 关联的事件条件

### 79.4 条件

默认去向：

- 编译为 `条件表`

多个条件统一归入同一 `condition_group_id`。

### 79.5 血量要求

默认去向：

- 编译为 `条件表`

它本质上是一类标准条件，只是策划书写时独立拆出。

### 79.6 触发几率

默认去向：

- 若技能行执行的是效果，则进入 `效果表.proc_chance`
- 若技能行执行的是修正，则进入 `修正表.proc_chance`

如果当前表结构没有 `proc_chance` 字段，建议补充。

### 79.7 技能效果

默认去向：

- 编译为 `效果表.effect_type + 参数字段`
- 或在修正规则线中编译为 `修正表.modify_field + modify_mode + modify_value`

---

## 80. 编译判定总规则

技能行编译时，先判断“这条技能行是在执行效果，还是在修改效果”。

建议按以下顺序判定：

1. 判断上下文是否为 `修正规则线`
2. 若不是，再判断技能效果语义是否属于“直接执行效果”
3. 若是直接执行效果，则编译到 `效果表`
4. 若带条件或血量要求，则额外生成 `条件表`
5. 若效果语义属于“修改现有技能字段”，则编译到 `修正表`

换句话说：

- `造成伤害`
- `附加减速`
- `获得护盾`
- `回复法力`

这类是“执行效果”，进 `效果表`

- `伤害加成+0.08`
- `冷却缩减0.20`
- `可穿透=true`
- `范围+0.12`

这类是“修正规则”，进 `修正表`

---

## 81. 编译到效果表的规则

当技能行表达的是“真正执行一次效果”时，编译到 `效果表`。

典型情况：

- 造成物理伤害
- 造成法术伤害
- 造成追加伤害
- 附加减速
- 附加冻结
- 获得护盾
- 回复法力
- 回复耐力
- 位移
- 叠加寒霜
- 引爆灼烧

### 81.1 效果表最小生成规则

每条技能行至少生成一条效果记录：

- `effect_id`
- `skill_id`
- `effect_type`
- `trigger_timing`
- `target_filter`
- `condition_group_id`

### 81.2 单条技能行生成多条效果记录

如果一条技能行中的技能效果语义包含多个标准效果，应拆成多条效果表记录。

例如：

```text
霜环 目标=周围敌人 效果=造成法术伤害0.8并附加减速0.20@2.0
```

应编译为：

- 一条 `DealDamage`
- 一条 `ApplyDebuff(slow)`

而不是一条记录塞两个效果。

---

## 82. 编译到条件表的规则

只要技能行中出现以下任一要素，就需要生成条件表记录：

- `条件`
- `血量要求`
- `触发器`（在条件触发线或修正规则线中）
- 某些特殊概率前置要求

### 82.1 条件组生成规则

一条技能行只生成一个主 `condition_group_id`。

该行内所有条件都归到同一组。

例如：

```text
破军追击 目标=命中目标 触发=命中时 血量=目标<0.35 条件=已激活破军核心 几率=1.0 效果=造成追加伤害0.8
```

可生成：

- 条件 1：事件 = 命中时
- 条件 2：目标生命比例 < 0.35
- 条件 3：已激活破军核心

三者同属一个 `condition_group_id`。

### 82.2 条件组与效果表的连接

效果表中的 `condition_group_id` 指向该条件组。

表示：

- 先判断条件组
- 再执行效果表记录

---

## 83. 编译到修正表的规则

当技能行表达的不是“直接施加效果”，而是“修改技能现有表现”时，编译到 `修正表`。

典型情况：

- 伤害加成 `+0.08`
- 冷却缩减 `0.20`
- 范围 `+0.12`
- 穿透开启
- 可追加追击
- 命中后刷新某技能冷却

### 83.1 修正语义识别规则

若技能效果语法满足以下任一特征，优先编译为修正：

- 明确修改现有字段
- 不产生独立命中结果
- 不产生独立 Buff / Debuff 实体
- 不直接造成一次独立伤害或位移

例如：

```text
流影回锋 条件=连续命中5次 效果=突进穿刺冷却缩减0.20
```

这不是新效果，而是在修改已有技能字段，应编译到修正表。

### 83.2 修正表最小生成规则

至少生成：

- `modifier_id`
- `source_type`
- `source_id`
- `target_skill_id`
- `modify_field`
- `modify_mode`
- `modify_value`
- `apply_condition_group_id`

---

## 84. 触发器的编译规则

触发器是技能行里最容易混淆的部分，必须单独定规则。

### 84.1 主动作线

主动作线中的技能行：

- 不允许填写 `触发器`
- 触发时机取自技能主时间线
- 编译时由编辑器或技能主配置提供 `trigger_timing`

### 84.2 条件触发线

条件触发线中的技能行：

- 允许填写 `触发器`
- `触发器` 应编译到条件表中的事件条件

例如：

- `触发=命中时` -> `condition_type = on_hit`
- `触发=暴击时` -> `condition_type = on_crit`
- `触发=格挡后` -> `condition_type = after_block`

### 84.3 修正规则线

修正规则线中的技能行：

- 触发器同样编译到条件表
- 修正表通过 `apply_condition_group_id` 关联该条件组

---

## 85. 血量要求的编译规则

血量要求虽然是技能行里的独立要素，但编译时必须转为标准条件。

映射建议：

- `血量=目标<0.35` -> `condition_type = target_hp_ratio_lt`, `param_1 = 0.35`
- `血量=目标>0.70` -> `condition_type = target_hp_ratio_gt`, `param_1 = 0.70`
- `血量=自身<0.50` -> `condition_type = self_hp_ratio_lt`, `param_1 = 0.50`
- `血量=自身>0.80` -> `condition_type = self_hp_ratio_gt`, `param_1 = 0.80`

血量要求永远不直接进入效果表或修正表本体字段。

---

## 86. 触发几率的编译规则

触发几率需要看技能行最终落到哪类表。

### 86.1 效果型技能行

若技能行编译到效果表：

- `几率` 进入效果表的 `proc_chance`

### 86.2 修正型技能行

若技能行编译到修正表：

- `几率` 进入修正表的 `proc_chance`

### 86.3 条件组中的概率

不建议把概率写进条件表。

原因：

- 概率不是布尔条件
- 它是条件成立后的额外判定步骤

因此建议流程固定为：

```text
先判断条件组
再判断 proc_chance
最后执行效果或修正
```

---

## 87. 技能效果语法到效果表的映射规则

这一层建议做标准映射表。

### 87.1 伤害类

```text
效果=造成物理伤害1.35
```

编译为：

- `effect_type = DealDamage`
- `damage_type = physical`
- `base_ratio = 1.35`

```text
效果=造成法术伤害0.8
```

编译为：

- `effect_type = DealDamage`
- `damage_type = magical`
- `base_ratio = 0.8`

### 87.2 追加伤害类

```text
效果=造成追加伤害0.8
```

编译为：

- `effect_type = DealExtraDamage`
- `extra_ratio = 0.8`

### 87.3 控制类

```text
效果=附加冻结@1.5
```

编译为：

- `effect_type = ApplyControl`
- `control_type = freeze`
- `control_duration = 1.5`

### 87.4 减益类

```text
效果=附加减速0.20@2.0
```

编译为：

- `effect_type = ApplyDebuff`
- `debuff_id = slow`
- `stack_count = 1`
- `param_1 = 0.20`
- `control_duration = 2.0`

### 87.5 回复类

```text
效果=回复法力0.03
```

编译为：

- `effect_type = RestoreMana`
- `heal_ratio = 0.03`

### 87.6 护盾类

```text
效果=获得护盾0.06@4.0
```

编译为：

- `effect_type = ApplyShield`
- `shield_ratio = 0.06`
- `control_duration = 4.0`

---

## 88. 技能效果语法到修正表的映射规则

以下写法优先视为修正语义。

### 88.1 冷却缩减

```text
效果=突进穿刺冷却缩减0.20
```

编译为：

- `target_skill_id = skill_spear_dash_001`
- `modify_field = cooldown`
- `modify_mode = mul`
- `modify_value = -0.20`

或统一解释为：

- `modify_mode = add_ratio`

具体模式你可以后面再定，但现在必须保证它进修正表，不进效果表。

### 88.2 范围扩大

```text
效果=霜环范围+0.12
```

编译为：

- `target_skill_id = skill_staff_frost_ring_001`
- `modify_field = radius`
- `modify_mode = mul`
- `modify_value = 0.12`

### 88.3 穿透开启

```text
效果=突进穿刺可穿透
```

编译为：

- `target_skill_id = skill_spear_dash_001`
- `modify_field = can_pierce`
- `modify_mode = override`
- `modify_value = true`

## 89. 单条技能行的编译案例

### 89.1 效果型案例

技能行：

```text
破军追击 目标=命中目标 触发=命中时 血量=目标<0.35 几率=1.0 效果=造成追加伤害0.8
```

编译结果：

#### 条件表

- 条件 1：`on_hit`
- 条件 2：`target_hp_ratio_lt = 0.35`

#### 效果表

- `effect_type = DealExtraDamage`
- `target_filter = hit_target`
- `extra_ratio = 0.8`
- `proc_chance = 1.0`
- `condition_group_id = cg_skill_pojun_chase_001`

#### 修正表

- 无

### 89.2 修正规则型案例

技能行：

```text
流影回锋 条件=连续命中5次 效果=突进穿刺冷却缩减0.20
```

编译结果：

#### 条件表

- 条件 1：`combo_hit_count_gte = 5`

#### 修正表

- `target_skill_id = skill_spear_dash_001`
- `modify_field = cooldown`
- `modify_mode = mul`
- `modify_value = -0.20`
- `apply_condition_group_id = cg_liuying_combo_001`

#### 效果表

- 无

## 90. 多效果技能行的拆分规则

如果策划写出复合型技能效果，编译器应主动拆分，而不是直接报废。

例如：

```text
霜环 目标=周围敌人 效果=造成法术伤害0.8并附加减速0.20@2.0
```

编译器应拆成：

- 效果 1：`DealDamage`
- 效果 2：`ApplyDebuff(slow)`

两条效果共用：

- 同一个 `skill_id`
- 同一个 `trigger_timing`
- 同一个 `condition_group_id`
- 同一个 `target_filter`

---

## 91. 编译优先级规则

为了避免歧义，编译器必须固定优先级。

建议优先级如下：

1. 先识别 `技能行上下文`
2. 再识别 `技能效果语义`
3. 再决定落到 `效果表` 还是 `修正表`
4. 再抽取 `条件 / 血量 / 触发器`
5. 最后附加 `目标选择器` 和 `触发几率`

关键原则：

- 上下文优先于文本表面形式
- “独立执行结果”优先归到效果表
- “修改现有字段”优先归到修正表

---

## 92. 编译错误处理规则

技能行编译时建议增加以下错误类型。

### 92.1 致命错误

- 缺少 `技能名`
- 缺少 `技能效果`
- 无法识别的技能效果语法
- 主动作线中非法填写 `触发器`

### 92.2 阻断错误

- 目标选择器无法识别
- 血量表达式非法
- 引用的目标技能不存在
- 修正规则没有目标技能

### 92.3 警告错误

- 省略目标选择器，自动回退默认目标
- 省略触发几率，自动补 `1.0`
- 复合效果被自动拆分

---

## 93. 第一版推荐编译器行为

如果第一版要尽快落地，建议编译器固定行为如下：

- 技能行先解析成 `SkillLineNode`
- 根据上下文分流到效果型或修正规则型
- `条件 / 血量 / 触发器` 一律先编译为条件组
- `技能效果` 再编译为效果表或修正表
- `触发几率` 不进条件表，只进效果表或修正表
- 复合效果自动拆分成多条效果记录

这套方式最容易让：

- 编辑器实现
- 配置调试
- 服务端读取

保持一致。

---

## 94. 技能行编译规则总结

技能行编译规则本质上是在解决一件事：

- 策划写的是“语义”
- 程序需要的是“结构”

所以必须固定转换关系：

- `条件 / 血量 / 触发器` -> `条件表`
- `独立执行效果` -> `效果表`
- `修改已有技能字段` -> `修正表`

把这层定清楚后，后续你再做：

- 技能行编辑器
- 技能导出工具
- 技能引擎读取器

都会顺很多。

## 113. 技能效果运行安全规则

为了防止技能效果失控，第一版需要固定安全限制。

### 113.1 递归限制

- `TriggerSkillLine` 最大深度：`3`
- 追加伤害不能继续触发追加伤害
- 召唤物触发技能必须受独立频率限制

### 113.2 数量限制

- 单次技能最大命中目标：按技能配置限制
- 单个角色同时存在召唤物：建议默认上限 `3`
- 单个角色同时存在区域效果：建议默认上限 `5`
- 单个角色同时存在光环：建议默认上限 `3`

### 113.3 频率限制

- 持续区域最小 tick 间隔建议 `0.5s`
- 光环刷新最小间隔建议 `0.5s`
- 引导射线最小 tick 间隔建议 `0.25s`

### 113.4 权威规则

- 伤害、控制、位移最终由服务端确认
- 客户端表现可以先播，但必须能接受服务端纠正

## 115. 技能效果设计总结

技能效果标准库的关键不是数量越多越好，而是每种效果的职责要稳定。

第一版建议坚持：

- 每个效果只做一类事情
- 复合技能由多个标准效果组合
- 伤害、控制、位移、召唤、光环、射线分别独立建模
- 属性修改和技能修正必须区分
- 表现辅助效果不能影响服务端权威结果

这样后续新增技能时，大多数需求都可以通过组合标准效果实现，而不是不断写特殊逻辑。

## 116. 参考 MythicMobs 后的技能行最终规范

本节作为第一版正式落地规范。若本节与前文旧版“技能行上下文分类”存在冲突，以本节为准。

参考 MythicMobs 的核心思想不是照搬怪物插件，而是保留它最稳定的三段式结构：

```text
Mechanic{参数} @Targeter{参数} ~Trigger{参数} ?Condition ?Condition
```

本项目是 2D RPG，因此删除以下不适合项：

- 删除 3D 高度、视线俯仰、飞行高度类参数
- 删除服务端指令、聊天、权限、仇恨表等插件专用效果
- 删除过度复杂的行上下文
- 删除脚本字符串条件
- 删除与本项目武器体系冲突的多元素武器配置

保留并强化以下概念：

- `技能组 SkillGroup`：一个技能由多条技能行组成
- `触发器 Trigger`：决定技能行何时执行
- `目标器 Targeter`：决定本行对谁生效
- `机制 Mechanic`：决定本行执行什么效果
- `条件 Condition`：决定本行是否允许执行
- `参数 Parameters`：所有数值必须量化

## 117. 技能组 SkillGroup

一个技能配置本质上是一个技能组。

```text
skill_id: skill_spear_dash_001
skill_group: combat.spear
lines:
  - damage{shape=line;range=4.5;width=0.6;max_targets=3;team=enemy;damage_type=physical;ratio=1.35;flat=0;can_crit=true} @s ~onCast{delay=0.18}
  - damage{damage_type=physical;ratio=0.8;flat=0;can_crit=false;extra_damage=true} @tg ~onHit ?target.target_hp_ratio<0.35 ?skill.talent:tianqu_pojun_core=1
```

规则：

- 一个技能组至少包含 1 条技能行
- 技能行按配置顺序读取，但执行顺序由 `Trigger + delay` 决定
- 同一技能组内允许多个技能行共享同一个触发器
- 技能组可以通过 `meta` 机制调用另一个技能组
- 技能组调用最大递归深度固定为 `3`

## 118. 技能行固定格式

技能行采用无序要素组合，但导出时统一格式化为：

```text
mechanic{key=value;key=value} @selector ~trigger{key=value} ?scope.condition:param>=value
```

组成规则：

- `mechanic{}` 必填，每行只能有 1 个主机制
- `@selector` 必填，缺省时由编辑器按机制类型补默认选择器
- `~trigger{}` 必填，主动技能主效果默认 `~onCast`
- `?condition` 可有 0 到多条，默认全部为 `AND`
- 多效果技能必须拆成多条技能行，不允许一行塞多个主机制

示例：

```text
move{mode=dash;distance=3.2;duration=0.15} @s ~onCast
damage{shape=line;range=4.5;width=0.6;max_targets=3;team=enemy;damage_type=physical;ratio=1.35;flat=0;can_crit=true} @s ~onCast{delay=0.18}
status{status_id=armor_break;operation=apply;value=0.12;duration=3} @tg ~onHit{chance=0.4}
```

## 119. 触发器 Trigger

第一版只保留战斗必要触发器：

| ID | 含义 | 常用场景 |
| --- | --- | --- |
| `onCast` | 释放时 | 主动技能起手、位移、生成区域 |
| `onHit` | 命中时 | 追加伤害、叠层、控制 |
| `onCrit` | 暴击时 | 暴击联动 |
| `onKill` | 击杀时 | 刷新冷却、回复资源 |
| `onTick` | 周期触发 | 光环、持续区域、引导射线 |
| `onDamaged` | 受击时 | 反击、护盾、被动防御 |
| `onDodge` | 闪避后 | 闪避反击、移速强化 |
| `onBlock` | 格挡后 | 盾反、减伤转伤害 |
| `onEnd` | 技能结束时 | 收尾爆炸、清理状态 |

触发器参数：

- `delay`：延迟秒数，默认 `0`
- `repeat`：重复次数，默认 `1`
- `interval`：重复间隔，默认 `0`
- `chance`：触发概率，范围 `0-1`，默认 `1`

示例：

```text
~onCast{delay=0.25}
~onTick{repeat=6;interval=0.5}
~onHit{chance=0.35}
```

## 120. 2D 锚点目标器 Targeter

本节是第三版目标选择器方案，已被第 137-142 节第四版选择器覆盖。

若两者冲突，以第四版为准。

第三版开始，目标器只描述“从哪里开始 / 以谁为引用”，不再描述命中形状。

必须拆清三层概念：

- `Targeter`：锚点或引用对象，例如自身、当前目标、命中目标、指定点。
- `Shape`：效果载体的几何形状，例如直线、扇形、圆形、射线、投射物碰撞体。
- `Filter`：命中过滤，例如敌我阵营、最大目标数、是否可重复命中。

因此不再使用：

```text
@line
@cone
@circle
@nearEnemies
@nearAllies
```

这些都是形状或过滤概念，应进入 Mechanic 参数。

| ID | 含义 | 必要参数 |
| --- | --- | --- |
| `self` | 以施法者为锚点 | 无 |
| `current` | 以当前锁定目标为锚点 | `range` |
| `hit` | 以本次命中目标为锚点 | 无 |
| `point` | 以指定落点为锚点 | `range` |
| `owner` | 以召唤物主人为锚点 | 无 |
| `marker` | 以场上标记点为锚点 | `range` |

目标器参数：

- `range`：搜索锚点的距离，单位米
- `max`：锚点数量上限，第一版通常为 `1`

形状参数进入 Mechanic，例如：

```text
damage{shape=line;range=4.5;width=0.6;max_targets=3;team=enemy;damage_type=physical;ratio=1.35} @s
damage{shape=cone;range=3.5;angle=70;max_targets=5;team=enemy;damage_type=physical;ratio=1.0} @s
damage{shape=circle;radius=3;max_targets=6;team=enemy;damage_type=magical;ratio=0.8} @zp
```

## 121. 条件 Condition

条件采用固定作用域：

| Scope | 含义 |
| --- | --- |
| `caster` | 施法者 |
| `target` | 目标 |
| `skill` | 当前技能或技能组 |
| `world` | 场景、地形、战斗环境 |

```text
?scope.condition:param operator value
```

导出时无空格：


第一版条件类型：

| ID | 类型 | 数值范围 |
| --- | --- | --- |

| `self_hp_ratio` | 自身生命比例 | `0-1` |
| `mana_ratio` | 法力比例 | `0-1` |
| `stamina_ratio` | 耐力比例 | `0-1` |
| `distance` | 双方距离 | `>=0` |
| `target_count` | 目标数量 | 整数 |
| `hit_count` | 连续命中数 | 整数 |
| `stack` | 指定层数 | 整数，需 `param` |

| `facing_angle` | 朝向夹角 | `0-180` |

| `terrain_tag` | 地形标签 | `0/1`，需 `param` |
| `state_tag` | 战斗状态标签 | `0/1`，需 `param` |

比较符固定为：

```text
>= <= > < = !=
```

## 125. 第一版安全限制

为防止技能组失控，固定以下限制：

- 单技能组最多 `16` 条技能行
- 单技能行只能有 `1` 个 Mechanic
- 多效果必须拆成多行
- `meta` 调用最大深度 `3`
- `onTick` 最小间隔 `0.25s`
- 单次目标上限 `max` 不得超过项目全局上限
- 追加伤害默认不再触发追加伤害
- 召唤物触发技能必须有独立冷却
- 客户端只做表现预测，服务端负责最终命中、伤害、控制和位移

## 126. 本次规范结论

最终技能系统不再使用复杂“行上下文”作为策划入口。

正式入口固定为：

```text
SkillGroup
 -> SkillLine
 -> Trigger
 -> Targeter
 -> Conditions
 -> Mechanic
 -> 结构化步骤表 / 效果表 / 条件表
```

这样既保留 MythicMobs 的组合效率，也符合本项目 2D RPG、武器附带技能、天赋强化技能的实际需求。

## 127. 条件库第二版

| Scope | 用途 |

| `mana_ratio` | `caster` | `ratio` | 否 | 法力门槛 |
| `stamina_ratio` | `caster` | `ratio` | 否 | 耐力门槛 |
| `distance` | `target` | `distance` | 否 | 近身、远程距离判断 |
| `target_count` | `skill` | `count` | 否 | 范围命中数量 |
| `hit_count` | `skill` | `count` | 否 | 连续命中、连段 |
| `stack` | `target` | `count` | 是 | 寒霜、灼烧、破甲层数 |
| `has_status` | `target` | `bool` | 是 | 目标是否有状态 |
| `offhand` | `caster` | `bool` | 是 | 副手联动 |
| `talent` | `skill` | `bool` | 是 | 天赋节点是否激活 |
| `cooldown_ready` | `skill` | `bool` | 是 | 指定技能是否冷却完成 |
| `facing_angle` | `target` | `angle` | 否 | 是否在正面、背后、扇形内 |
| `line_of_sight` | `world` | `bool` | 否 | 2D 障碍阻挡 |
| `terrain_tag` | `world` | `bool` | 是 | 地形联动 |
| `state_tag` | `caster` | `bool` | 是 | 格挡、引导、浮空等状态 |
| `resource_value` | `caster` | `number` | 是 | 生命/法力/耐力绝对值 |
| `random` | `skill` | `ratio` | 否 | 特殊随机判定 |

### 127.4 条件写法示例

```text
?target.target_hp_ratio<0.35
?caster.stamina_ratio>=0.2
?target.stack:frost>=4
?target.has_status:slow=1
?skill.talent:tianqu_pojun_core=1
?world.line_of_sight=1
```

### 127.5 条件组

每个技能步骤拥有 0 或 1 个主条件组。

第一版条件组只支持：

```text
logic = AND
```

如果策划需要 OR：

- 编辑器拆成多条技能行
- 每条技能行拥有独立条件组
- 不允许在同一条件组里嵌套复杂布尔表达式

这样做的原因是运行时更稳定，策划调试也更直观。

## 129. 编辑器导出第二版

编辑器导出结构固定为：

```text
skill
skill_steps
skill_mechanics
skill_condition_groups
skill_conditions
skill_text
issues
```

### 129.1 skill_steps 新增字段

```json
{
  "condition_group_id": "cg_skill_spear_dash_001_2",
  "interval": 0
}
```

说明：

- `condition_group_id` 为空表示该步骤无条件
- `interval` 用于重复触发

### 129.2 skill_mechanics 结构

效果表不再把参数摊平到顶层，而是固定为：

```json
{
  "mechanic_id": "mech_skill_spear_dash_001_1",
  "step_id": "step_skill_spear_dash_001_1",
  "skill_id": "skill_spear_dash_001",
  "mechanic_type": "damage",
  "mechanic_params": {
    "damage_type": "physical",
    "ratio": 1.35,
    "flat": 0,
    "can_crit": true
  }
}
```

这样服务端可以按 `mechanic_type` 分发执行器，参数保持原样读取。

### 129.3 skill_condition_groups

```json
{
  "condition_group_id": "cg_skill_spear_dash_001_2",
  "step_id": "step_skill_spear_dash_001_2",
  "skill_id": "skill_spear_dash_001",
  "logic": "AND"
}
```

第一版只支持 `AND`。

### 129.4 skill_conditions

```json
{
  "condition_id": "cond_skill_spear_dash_001_2_1",
  "condition_group_id": "cg_skill_spear_dash_001_2",
  "step_id": "step_skill_spear_dash_001_2",
  "skill_id": "skill_spear_dash_001",
  "condition_scope": "target",
  "condition_type": "target_hp_ratio",
  "compare_operator": "lt",
  "value": 0.35,
  "param": ""
}
```

### 129.5 编辑器校验规则

编辑器至少校验：

- 条件数值是否在范围内
- 条件是否缺少必填参数
- 条件比较符是否被当前条件支持
- 锚点目标器是否缺少搜索距离
- Mechanic 形状是否缺少必要距离、半径、角度或宽度
- 持续效果是否缺少 `duration` 或 `tick`
- 投射物是否缺少速度或射程
- 召唤数量是否超过上限
- 陷阱是否缺少触发技能
- `meta` 是否超过递归深度上限

## 130. 第二版优化结论

当前技能系统的设计重点已经从“列出很多效果名”转为“每个条件和效果都能被编辑器约束、被服务端读取、被策划调试”。

后续继续扩展时，应优先增加：

- 条件元数据
- Mechanic 参数定义
- 校验规则
- 编译输出字段

不要优先增加无法校验的自由文本逻辑。

## 131. 接近 MythicMobs 的 Mechanic 第三版规范

本节覆盖第二版 Mechanic 清单。第三版目标不是继续堆效果数量，而是让效果系统接近 MythicMobs 的组合体验，同时保持 2D RPG 可用。

MythicMobs 值得保留的能力：

- 每条技能行只做一个明确 Mechanic
- Mechanic、Targeter、Condition 分离
- 复杂效果通过多条技能行和 MetaSkill 组合
- Delay、Repeat、Chance 属于技能行调度，不写进 Mechanic 本身
- Mechanic 参数短、稳定、可校验

本项目不保留的能力：

- 聊天、命令、权限、仇恨表、怪物 AI 专用效果
- 3D 高度、飞行、视线俯仰
- 任意脚本执行
- 同一行内写多个主效果
- 无法服务端校验的自由文本逻辑

## 132. Mechanic 分类

第三版 Mechanic 按玩法用途分 8 类。

| 分类 | 用途 |
| --- | --- |
| `damage` | 伤害、治疗、持续伤害 |
| `shape` | 投射物、射线、区域、光环、环绕物 |
| `motion` | 位移、速度、力场 |
| `state` | 状态、控制、护盾、免疫、驱散、资源、属性、层数 |
| `entity` | 召唤、陷阱、标记点 |
| `logic` | 冷却、连击、连锁、事件 |
| `meta` | 技能组调用 |

编辑器必须按分类筛选，否则后期效果数量上来后不可用。

## 135. 第三版可用性要求

编辑器必须做到：

- Mechanic 按分类筛选
- 显示每个 Mechanic 的推荐锚点目标器
- 使用不推荐锚点目标器时给警告

- 调用类效果必须校验 `skill_ref` 或 `effect_ref`
- 递归和重复触发必须有硬上限

服务端必须做到：

- 按 `mechanic_type` 分发执行器
- 不执行未知 Mechanic
- 不执行参数非法 Mechanic
- 不信任客户端导出的命中结果
- 所有锚点目标器和 Mechanic 形状重新计算
- 所有伤害、控制、位移最终由服务端确认

## 136. Targeter 与 Shape 纠偏结论

旧版文档中出现过 `目标=前方直线`、`@line`、`@cone`、`@circle` 这类写法。

这些写法在第三版以后视为历史示例，不再作为正式规范。

正式规范固定为：

```text
Targeter = 锚点
Shape = Mechanic 参数
Filter = Mechanic 参数
```

正确：

```text
damage{shape=line;range=4.5;width=0.6;max_targets=3;team=enemy;damage_type=physical;ratio=1.35} @s
area{shape=circle;radius=3;duration=4;tick=1;team=enemy} @zp
ray{length=8;width=0.5;damage_type=magical;ratio=1.0;max_targets=5} @s
```

错误：

```text
damage{damage_type=physical;ratio=1.35} @line{range=4.5;width=0.6;max=3}
damage{damage_type=physical;ratio=1.0} @cone{range=3.5;angle=70;max=5}
damage{damage_type=magical;ratio=0.8} @circle{radius=3;max=6}
```

原因：

- 直线、扇形、圆形是命中形状，不是目标来源
- 敌方、友方、最大数量是命中过滤，不是目标来源
- 目标器只负责确定技能从哪个对象或点开始计算
- 服务端可以按 `Targeter -> Shape -> Filter -> Mechanic` 的顺序稳定执行

## 137. 目标选择器第四版

第四版采用更接近 MythicMobs 的选择器表达式，但按 2D RPG 玩法重新命名和收敛。

第四版比第三版更好，原因是：

- 表达力更强：能表达实体、坐标、玩家查询、实体查询、多边形、圆形、扇形、椭圆。
- 语法统一：所有目标来源都以 `@` 开头，策划可以直接阅读技能行。
- 可扩展：后续新增变量查询、标记查询、阵营过滤时不需要增加大量下拉枚举。
- 更接近 MythicMobs：保留 `@selector(args)` 的技能文本习惯。

但它也有代价：

- 语法更复杂，编辑器必须提供模板和校验。
- 范围选择器不应滥用，复杂多边形需要服务端做性能限制。
- `@p/@e` 查询变量必须白名单化，不能允许任意脚本查询。

因此最终建议：

- 策划界面使用模板生成。
- 导出文本使用完整选择器表达式。
- 服务端只执行解析后的结构化 selector AST。

## 138. 单实体选择器

单实体选择器返回一个实体。

| 选择器 | 含义 |
| --- | --- |
| `@s` | 自身 |
| `@t` | 指向目标 |
| `@tg` | 触发目标 |
| `@np` | 最近玩家 |
| `@ne` | 最近实体 |
| `@ow` | 主人 |
| `@p(n=x)` | 名称为 `x` 的玩家 |
| `@p(id=x)` | ID 为 `x` 的玩家 |
| `@p(%a=x)` | 变量 `a` 为 `x` 的玩家 |
| `@e(n=x)` | 名称为 `x` 的实体 |
| `@e(id=x)` | ID 为 `x` 的实体 |
| `@e(%a=x)` | 变量 `a` 为 `x` 的实体 |

示例：

```text
damage{shape=target;damage_type=physical;ratio=1.0} @t
status{status_id=slow;operation=apply;value=0.2;duration=2} @tg
heal{ratio=0.8;flat=0} @p(n=队友A)
```

## 139. 单坐标选择器

单坐标选择器返回一个坐标。

| 选择器 | 含义 |
| --- | --- |
| `@slo` | 自身坐标 |
| `@tlo` | 指向目标坐标 |
| `@tglo` | 触发目标坐标 |
| `@owlo` | 主人坐标 |
| `@pl(p=x,y,z)` | 指定坐标 |
| `@zp` | 指向坐标，来自鼠标或摇杆 |

`@zp` 规则：

- 如果指向距离小于最远范围，使用真实指向坐标。
- 如果超出最远范围，坐标钳制到最远范围位置。
- 最远范围由技能步骤的 `targeter_params.range` 或技能基础射程提供。

示例：

```text
area{shape=circle;radius=3;duration=4;tick=1;team=enemy} @zp
trap{trap_id=spike_trap;trigger_radius=1.2;duration=10;effect_ref=skill_trap_spike} @pl(p=10,4,0)
```

## 140. 范围选择器

范围选择器返回一个区域或区域内目标集合。

### 140.1 多边形边界

```text
@bj=(@slo/@zp/@pl(p=10,4,0))
```

含义：

- 多个单坐标组成多边形边界
- 支持 `width` 宽度参数
- 适合墙、刀光路径、封锁线

建议格式：

```text
@bj=(@slo/@zp/@pl(p=10,4,0);w=0.4)
```

### 140.2 多边形范围

```text
@fw=(@slo/@pl(p=4,0,0)/@pl(p=4,4,0)/@pl(p=0,4,0))
```

含义：

- 多个单坐标组成封闭多边形范围
- 适合不规则区域、地形技能、封印区

### 140.3 圆形

```text
@slo+o(r=3)
@pl(p=10,4,0)+o(r=2)
```

含义：

- 单坐标为圆心
- 半径为 `r`

### 140.4 扇形

```text
@slo+o(r=4,ag=90,c=70)
@slo+o(r=4,ag=90,ctc=70)
```

含义：

- 单坐标为圆心
- 半径为 `r`
- 一条边方向为 `ag` 度
- `c=q` 表示顺时针 `q` 度
- `ctc=q` 表示逆时针 `q` 度

### 140.5 椭圆

```text
@slo/@zp+o(r=2)
```

含义：

- 两个单坐标形成椭圆主轴
- `r` 表示副轴半径或宽度

## 141. 新目标选择器与第三版方案对比

第四版更好，但前提是必须由编辑器辅助生成。

| 维度 | 第三版锚点目标器 | 第四版选择器 |
| --- | --- | --- |
| 易学性 | 更简单 | 更复杂 |
| 表达力 | 中等 | 高 |
| 接近 MythicMobs | 中等 | 更高 |
| 编辑器依赖 | 低 | 高 |
| 服务端解析成本 | 低 | 中 |
| 是否适合正式玩法 | 适合原型 | 更适合长期维护 |

结论：

- 原型期可以用第三版。
- 正式玩法应使用第四版。
- 第四版必须限制变量查询、区域复杂度和最大目标数。

## 142. 第四版选择器安全限制

服务端必须限制：

- `@p/%a` 和 `@e/%a` 只能查询白名单变量。
- `@bj` 和 `@fw` 最大坐标点数建议不超过 `8`。
- 范围选择器必须有最大目标数。
- `@zp` 必须受技能最大射程限制。
- 复杂范围每次技能释放只解析一次，不能每帧重新解析。
- 选择器解析失败时技能步骤不执行，并输出明确错误。
