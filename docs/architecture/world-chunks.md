# 世界区块与 Rust 地图生成设计

## 当前实现范围

当前仓库已落地的服务端基础：

- 每个区块 `80x80` tile
- 每张地图边界为 `6400 x 6400` 区块
- 玩家附近仅加载 `5x5` 区块窗口
- 超出当前地图边界时切换到新 `mapId`
- 每张地图按目录单独保存区块文件
- 区块卸载时自动持久化
- 区块存档已预留 `deltaTiles` 结构，用于只保存玩家改动

## 核心原则

区块不是“地形类型锁定单元”，而是：

- 加载单元
- 卸载单元
- 持久化单元
- 激活模拟单元

地形、生物群系、资源点必须按 **全局世界坐标** 连续生成。

当前协议层已开始拆分为：

- `terrain`：地形基底
- `feature`：地表特征或入口
- `levelHint`：层级入口提示

当前实体坐标规则：

- 协议里的玩家 `position` 表示实体中心
- 玩家逻辑占地固定为 `1x1` 格
- 区块归属、脚下地形、事件触发格等均按实体中心落点换算

地图表现层未来应基于这些数据生成真正的地图资源，而不是把颜色块本身当地图。

这意味着：

- 不能出现“一个区块只有一种地形颜色”的硬切块效果
- 区块边界不应成为地形边界
- 跨区块、跨地图分页时，地表变化应像同一张连续世界

## 职责划分

### Go 服务端负责

- 玩家会话
- 当前位置与 `mapId` 管理
- 5x5 区块窗口加载
- 区块卸载与保存
- WebSocket / HTTP 同步
- 存档目录组织

### Rust 负责

- 按全局坐标生成连续地形内容
- 地图边界连续性控制
- 生物群系、地表、资源点分布
- 后续高性能噪声与地形规则计算
- 洞口、潜水口、山口等地形入口标记生成

## 地图连续性策略

为了避免“跨地图割裂感”，地图生成不能把每张地图当成孤岛。

当前设计要求：

- 生成输入必须包含全局世界坐标信息
- `mapId` 只是存档分页，不是世界断点
- Rust 生成器应基于全局种子和全局坐标连续采样
- 这样玩家跨出一张地图时，只是切到新的存档分页，视觉上仍然连续

## 正确的生成模型

应当参考 Minecraft 这类连续世界：

- 先根据全局坐标生成连续噪声场
- 再从噪声场推导湿度、温度、海拔、侵蚀等参数
- 最后映射到具体地表类型、植被、岩石、水域

区块只负责把这片连续世界按窗口切出来。

因此正确公式应是：

- `tile = f(world_seed, world_x, world_y, rules)`

而不是：

- `chunk_type = f(chunk_x, chunk_y)` 后整块铺同一种地形

## 当前地形目标

当前生成器应逐步覆盖以下地形/入口类型：

- 平原
- 雪原
- 热带草原
- 阔叶林
- 针叶林
- 沙漠
- 戈壁
- 湖泊
- 湖底
- 河流
- 海洋
- 热带海洋
- 冰洋
- 高山
- 高原
- 山洞入口
- 地下洞穴入口
- 沙滩
- 海底

当前阶段说明：

- 已先实现这些类型的服务端生成标记与客户端调试颜色区分
- `z=1 / z=0 / z=-1` 的真正层切换交互仍需后续补齐

## 持久化目录

```text
server/data/worlds/
  map_0_0/
    chunk_0_0.json
    chunk_0_1.json
  map_1_0/
    chunk_0_0.json
```

## 当前接口

- `GET /api/v1/world/chunks?token=...`
  - 返回玩家当前附近 `5x5` 区块
  - 返回已卸载区块列表
  - 返回地图切换信息

## 当前限制

- 当前 Rust 生成器已可执行，但客户端仍使用占位渲染而非正式 Tilemap 美术层
- 当前 `deltaTiles` 结构已预留，但建造/破坏写入逻辑尚未接入
- 当前客户端仍是采样点方块渲染，尚未达到正式 Tilemap 精度
